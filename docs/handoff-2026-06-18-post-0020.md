## Next sequential item

### 0021 — End meeting, final pass, review & edit, Discussion Summaries

**Depends on:** 0008 (extraction loop scheduler, final pass), 0018 (live runtime).

**Goal:** Stop → final extraction pass → review screen showing per-Agenda-Item Discussion Summary + Decisions + Actions, all editable.

**Full spec in [BACKLOG.md](../BACKLOG.md) at line 269.**

Key things to know before starting:

1. **"End meeting" IPC channel does not exist yet.** There is no `meeting:end` invoke channel. It needs to be added to `src/shared/ipc.ts` (both the schema and the `IpcChannel` union), wired in `src/main/ipc-registry.ts`, added to `IPC_CHANNELS` in `src/main/index.ts`, and exposed on `window.api` via `src/preload/index.ts`.

2. **The placeholder meeting wiring.** `src/main/index.ts` uses `PLACEHOLDER_MEETING_ID = 'active-session'` and constructs `LiveExtractionRuntime` with empty agenda/participants. Item 0021 replaces this with a real persisted meeting. `endMeeting()` on the runtime must be called with the real `Meeting` object (state `'ended'`). `MeetingLifecycleService` enforces the `Draft → Live → Ended` transition — use it.

3. **`endMeeting()` already exists on `LiveExtractionRuntime`** (from item 0018). It: (a) calls the scheduler's final pass, (b) emits `items:summaries` with Discussion Summaries, (c) stops the runtime. The review screen consumes that `items:summaries` event. The `onItemsSummaries` subscription in LiveScreen currently has an empty handler (`// Post-meeting summaries handled in item 0021 Review screen`) — that's the hook.

4. **`ReviewScreen.tsx` exists but is empty.** It was stubbed in item 0013. It needs to be populated: render per-Agenda-Item Discussion Summaries + Decisions + Actions, all editable (same edit flow as Live screen for items, but also for summaries). Ended meetings remain fully editable.

5. **`MeetingLifecycleService`** is in `src/main/services/meetingLifecycleService.ts`. It has `startMeeting()` and presumably needs an `endMeeting()` or it already exposes it. Read it before wiring.

6. **Discussion Summaries** are stored in `discussionSummaryRepo` (SQLite, `src/main/db/repos/`). They arrive via `items:summaries` IPC push after the final pass. The review screen should load them from the store (or request them via an IPC fetch after navigation).

7. **Running summary vs Discussion Summary** — make sure the boundary stays clean (CONTEXT.md): Discussion Summaries are final-pass per-Agenda-Item; Running Summary is live in-memory and not part of the deliverable.
