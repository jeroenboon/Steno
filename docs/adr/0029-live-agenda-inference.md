# Live agenda inference

Until now the agent only ever proposed Decisions and Actions during Live; the agenda was a fixed spine the note-taker set up in Draft, and CONTEXT.md stated "agenda never groomed live". To support meetings that start without preparation (the agenda is spoken at the top, or never written down), the agent now also infers Agenda Items mid-meeting on a slow cadence and proposes them, and `AgendaItem` gains the same Proposed/Confirmed lifecycle as Decisions and Actions. User-entered or pasted Draft items are Confirmed; live- and import-inferred items are Proposed until a human accepts them.

## Decisions

- **Separate slow cadence.** Live agenda inference runs on its own scheduler, distinct from the ~15-30s extraction cadence, so it doesn't spend tokens re-deriving the agenda every extraction turn.
- **Append-only with grounding.** Each slow tick passes the current agenda as grounding context and the provider returns only topics not already covered. The agent does **not** revise or retract its own live-proposed agenda items; a rough guess lingers as Proposed until the note-taker edits or dismisses it, and the final pass re-infers cleanly. A normalised-title dedup guard catches near-duplicates.
- **Routing asymmetry — live is conservative, the final pass is complete.** During Live, only Confirmed Agenda Items (plus the built-in Off-agenda bucket) are valid routing targets for Decisions and Actions; a Proposed agenda item becomes a bucket only once the note-taker confirms it. This keeps the live spine stable under the note-taker and avoids dangling references when a Proposed item is dismissed. The **final pass** routes into the full agenda (Confirmed + Proposed/inferred) and regroups the Off-agenda items beneath it. Items already assigned live are never silently re-bucketed; the final pass is the authority for the definitive grouping.
- **One generalised `inferContext`.** The same port method serves paste (Draft, free text), live (transcript window + grounding), and the final pass (whole transcript). It takes a source (`{ text }` or `{ spans }`) and optional `knownAgendaItems` for grounding, and returns `InferredContext` (now with an optional `title`). Pasting in Draft is an input method, not inference: it structures the user's own text into Confirmed items. Live inference covers Agenda Items only; Participants are inferred at the final pass (and via paste/import), not on the live tick.

## Considered options

- **Reconciling instead of append-only**: provider returns the full agenda each tick, runtime fuzzy-matches to revise/retract its own Proposed items. Rejected for V1: a matching layer (same class of problem as owner/agenda assignment) and ID churn/flicker, for stability the final pass already provides.
- **Routing live into Proposed agenda items**: faster "magic", but introduces dangling-reference handling and an unstable spine under the note-taker. Rejected.
- **A dedicated live port method** instead of generalising `inferContext`: more explicit delta contract, but a third method covering the same ground and a second prompt to maintain. Rejected in favour of the smaller port surface.

## Consequences

- Forward-only migrations: a `state` column on `agenda_items` (default `confirmed`, so existing rows are unaffected) and a `titleAutoGenerated` flag on `meetings` for the quick-start auto-title (final pass replaces the placeholder only while the flag is set; a user-edited title is never overwritten).
- The owner/agenda assignment must treat only Confirmed agenda items as live routing targets, and all agenda items at the final pass.
- Import stays correct: its inferred agenda (Proposed) is routable because the final pass uses the full agenda, not Confirmed-only.

## Implementation notes (as built)

- **Slow cadence default: 90 s** (`AgendaInferenceScheduler`, constructor-tunable). Driven by the same injected `Clock` as the extraction loop; `pause()`/`resume()` mirror the meeting pause, and `endMeeting()` pauses it before the final pass so no late tick races finalisation.
- **Renderer event: a dedicated `agenda:changed` push** carrying the full current agenda (Confirmed + Proposed), chosen over overloading `items:changed` (smaller conceptual diff). The scheduler exposes an `onProposed` callback the `LiveExtractionRuntime` uses to emit it. Grooming uses `agendaItem:confirm` and `agendaItem:editAndConfirm` invoke channels (dismiss reuses `agendaItem:remove`), persisting through `agendaItemRepo.update` so a confirmed item is a live routing bucket on the next tick.
- **Final-pass inference is gated to `meeting.source === 'live'` with an empty agenda.** Import infers itself before building the runtime, so the runtime-end inference would otherwise double up. The runtime takes optional agenda/participant/meeting repos; absent ⇒ inference is skipped.
- **The normalised-title compare is shared** (`src/shared/agenda/agendaTitle.ts`: `normaliseAgendaTitle`, `isTitleCovered`, `excludeCoveredAgendaItems`) between the adapter grounding filter and the proposal-service dedup, so the two paths can never drift.
- Live agenda inference covers Agenda Items only; the live tick ignores `title`/`participants` even though `inferContext` can return them.
